title: "Platform API"
description: "Manage Wazoo organizations, worlds, tokens, and usage."

The Wazoo Platform API is the public-facing management API for hosted Wazoo
resources at `api.wazoo.dev`. It is separate from the Worlds Data API at
`api.worlds.wazoo.dev` and the human Console at `console.wazoo.dev`.

Use the Platform API to create and inspect beta applications, organizations,
world metadata, platform tokens, world auth tokens, usage records, limits, and billing state.
Use the Worlds Data API for graph operations such as search, SPARQL, import, and
export.

<Info>
  The current Platform API reference is implemented as a route list at
  `/openapi.json`. A complete schema-level OpenAPI document is not published
  yet.
</Info>

## Authentication

Platform API requests use platform tokens with the `wzp_` prefix.

```http
Authorization: Bearer wzp_...
```

Platform tokens can manage resources. They are not the same as world auth
tokens. World auth tokens use the `wzw_` prefix and are intended for data-plane
access to specific worlds.

## Base URL

```text
https://api.wazoo.dev/v1/
```

For local development of the Platform API server, use the Wrangler dev URL for
your Worker.

## Current resources

The Platform API currently covers these management resources:

| Resource          | Routes                                                                                               |
| ----------------- | ---------------------------------------------------------------------------------------------------- |
| Organizations     | `/v1/organizations`, `/v1/organizations/:organizationId`                                             |
| Worlds            | `/v1/organizations/:organizationId/worlds`                                                           |
| Platform tokens   | `/v1/auth/api-tokens`, `/v1/organizations/:organizationId/platform-tokens`                           |
| World auth tokens | `/v1/organizations/:organizationId/worlds/:worldId/auth/tokens`                                      |
| Usage             | `/v1/organizations/:organizationId/usage`, `/v1/organizations/:organizationId/worlds/:worldId/usage` |
| Limits            | `/v1/organizations/:organizationId/limits`                                                           |
| Billing           | `/v1/organizations/:organizationId/billing`, `/v1/organizations/:organizationId/billing:openPortal`   |
| Beta applications | `/v1/betaApplications`, `/v1/betaApplications/:applicationUid:approve`                                |
| Stripe webhook    | `/v1/stripe/webhook`                                                                                  |

`organizationId` path parameters accept either the public organization ID, such
as `acme`, or the canonical resource name `organizations/acme`.

Public `organizationId` and `worldId` values must match:

```text
^[a-z][a-z0-9-]{2,62}$
```

Internal `uid` values intentionally use prefixes such as `org_...` and `w_...`
so they cannot be confused with public path IDs.

The public API follows AIP-style naming:

```text
name = organizations/{organization} or organizations/{organization}/worlds/{world}
uid = output-only internal support ID, such as org_... or w_...
displayName = mutable human label
```

The control schema mirrors that model with `organization_id`, `world_id`, and
`display_name` columns instead of ambiguous `slug` or human-name fields.

Groups are postponed for v1. The v1 hierarchy is:

```text
organizations/{organization}
organizations/{organization}/worlds/{world}
```

Intra-organization projects, folders, groups, tags, or collections can be added
later if they carry distinct policy, quota, or organization behavior.

Organization and World deletes are soft-retained. Delete records `deleteTime`,
`expireTime`, and internal purge state; list endpoints exclude deleted resources
by default, while direct get can return retained deleted resources until purge.

World lifecycle routes include:

```http
DELETE /v1/organizations/:organizationId/worlds/:worldId
POST /v1/organizations/:organizationId/worlds/:worldId:undelete
POST /v1/organizations/:organizationId/worlds/:worldId:sync
```

World delete revokes world auth tokens. World undelete runs reconciliation before
reactivating the World and does not restore revoked tokens.

## Data-plane boundary

The Platform API does not execute SPARQL, search triples, import RDF, or export
RDF. Those operations belong to the Worlds Data API and `@worlds/client`.

The intended production boundary is:

- Wazoo Console runs at `console.wazoo.dev` for human beta application and
  management flows.
- Wazoo Platform API runs at `api.wazoo.dev` for management/control-plane calls.
- Worlds Data API runs at `api.worlds.wazoo.dev` for graph/search/SPARQL/import/export.
- `wzp_` platform tokens manage organizations, worlds, tokens, and usage.
- `wzw_` world tokens authorize world data access.
- Data-plane services can integrate with platform usage and limits, but that
  enforcement path is not fully documented yet.

## Beta applications

Private beta applications are public submissions protected by Cloudflare
Turnstile. Review actions require a manually seeded global admin platform token.

```http
POST /v1/betaApplications
GET /v1/betaApplications?state=PENDING
POST /v1/betaApplications/:applicationUid:approve
POST /v1/betaApplications/:applicationUid:reject
```

Public submission accepts:

```json
{
  "email": "founder@example.com",
  "applicantName": "Ada Lovelace",
  "company": "Example Co",
  "useCase": "Hosted Worlds for support agents",
  "turnstileToken": "..."
}
```

Approval creates an Organization only. It does not create a starter World.

```json
{
  "organizationId": "example-co",
  "displayName": "Example Co",
  "reviewNote": "Approved for private beta"
}
```

## Usage and limits

Wazoo usage and limits use transparent resource accounting. Data-plane services
should report the underlying resources consumed by world operations so the
Platform API can aggregate usage and enforce limits.

The production model is:

- Worlds Data API operations consume resources such as database operations,
  import bytes, query duration, storage bytes, and egress bytes.
- The Platform API records and aggregates integer resource usage.
- Entitlements and limits decide whether an organization or world can continue
  running work.
- If a configured resource limit would be exceeded, the API returns
  `RESOURCE_EXHAUSTED` with a quota object.

Private beta usage is recorded for accounting and entitlement evaluation only.
No credits are deducted and no live payment enforcement occurs. When provider
cost is known, Wazoo records it as an estimate; when it is unknown, cost fields
stay empty rather than pretending precision.

Usage events include:

```text
name
organization
world
metric
quantity
unit
providerCostMicrocents
wazooMarkupMicrocents
estimatedCostMicrocents
billingSource
createTime
```

During private beta, quota policies are code-defined and resource-based. The
default policy starts with a maximum active Worlds ceiling and can expand as
data-plane resource meters come online.

Effective quota states are:

```text
OK
WARN
THROTTLED
SUSPENDED
MANUAL_REVIEW
```

Management gates in v1:

| Operation          | OK/WARN | THROTTLED                  | SUSPENDED                  |
| ------------------ | ------- | -------------------------- | -------------------------- |
| Create World       | Allowed | Blocked                    | Blocked                    |
| Sync World         | Allowed | Allowed with `worlds.admin`| Admin/support only         |
| Undelete World     | Allowed | Allowed if within limits   | Blocked except admin       |
| Create World token | Allowed | Blocked                    | Blocked                    |
| Revoke World token | Allowed | Allowed                    | Allowed                    |
| Rotate World token | Allowed | Revoke-and-replace only    | Blocked                    |

The Platform API records management-plane usage for hosted Worlds operations:

```text
world.create.count
world.provision.count
world.sync.count
```

Usage reads return aggregate totals plus a recent event ledger, including unit,
billing source, and optional provider-cost estimate fields.

## Provisioning status

World responses expose both product lifecycle state and infrastructure state.
New Worlds are provisioned as one Turso/libSQL database per World. During
creation, the Platform API creates the Turso database, mints a short-lived
database auth token, initializes the World schema metadata, and then marks
`provisioning.state = "ACTIVE"`.

The control DB stores each World's deterministic Turso database name. It is
derived from the Wazoo environment and immutable World `uid`, then reused by
provisioning and sync rather than rediscovered from mutable public IDs.

`POST /v1/organizations/:organizationId/worlds/:worldId:sync` returns a
`syncReport`. Sync verifies that the deterministic Turso database exists and
re-initializes the safe schema/identity metadata when needed.

Sync report statuses are `HEALTHY`, `REPAIRED`, `BLOCKED`, and `FAILED`.

## Billing state

Billing is a production requirement for the Platform API. Stripe is the source
of truth for billing and subscription state.

Wazoo should not invent its own subscription ledger. The Platform API should use
Stripe for customer, subscription, payment, and lifecycle state, then mirror
only the entitlement data needed for product behavior and limit enforcement.

Current beta billing routes expose Stripe-shaped state without payment
enforcement:

```http
GET /v1/organizations/:organizationId/billing
POST /v1/organizations/:organizationId/billing:openPortal
POST /v1/stripe/webhook
```

Most private beta organizations return `state = "BETA_FREE"`,
`provider = "STRIPE"`, and `paymentRequired = false`. Portal creation returns a
clear precondition error until a Stripe customer exists.

## Admin tokens and audit

Global admin platform tokens are manually seeded. They use `kind = "ADMIN"`,
`organization_uid = NULL`, and include the `admin` scope. Public token creation
endpoints cannot mint admin tokens.

Admin-only actions and admin quota bypasses write internal audit events. There
is no public audit API in v1.

## World token rotation

World auth token rotation is an atomic revoke-and-replace operation. Under
`THROTTLED`, rotation is allowed only when it revokes existing World tokens and
returns one replacement without expanding access duration or count.
