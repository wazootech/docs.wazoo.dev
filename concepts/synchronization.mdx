---
title: "Synchronization"
description: "Atomic mutation queues and reliable persistence."
---

Synchronization is the process by which atomic mutation queues (inserts and deletes) generated by application interactions are serialized and reliably persisted to authoritative storage providers.

## Mutation pipeline

```
Application mutation
        │
        ▼
┌─────────────────┐
│ Idempotency     │── Evaluates existing state before emitting signals
│ Guard           │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ Mutation Queue  │── Atomic insert/delete operations
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ Storage Adapter │── LibSQL / Deno KV / In-memory
└─────────────────┘
```

## Idempotency guards

The idempotency guard layer evaluates existing graph store state **before** emitting mutation signals. This:

- Suppresses redundant external API interactions.
- Prevents database duplication.
- Saves compute and API costs (especially for LLM embedding services).

## Dual-layer safety

1. **In-memory checks first** — Fast `.has()` lookups suppress redundant downstream calls.
2. **Storage-level integrity** — Pre-emptive destructive wipes or atomic conflict resolutions at the SQL level prevent dirty state across reboots.
